# Organização Recomendada do GitHub

Este documento define como o GitHub deve ser organizado para funcionar como portfólio profissional e base pública de evolução técnica.

---

## Categorias principais

### Projetos Principais

Repositórios que devem ficar em destaque no perfil.

- `sacramento-os`
- `debee`
- `linx-dms-knowledge-base`
- `support-lab`
- `automation-scripts`

---

## Laboratórios

Projetos práticos para demonstrar evolução técnica.

- `lab-network`
- `lab-active-directory`
- `lab-microsoft365`
- `lab-sql`
- `lab-troubleshooting`
- `lab-noc`

Cada laboratório deve conter:

```text
README.md
docs/
exercises/
notes/
diagrams/
```

---

## Estudos

Repositórios para organizar fundamentos e anotações práticas.

- `tcp-ip-fundamentals`
- `windows-support`
- `active-directory-study`
- `microsoft365-study`
- `itil-notes`
- `noc-study`
- `servicenow-study`

## Documentação Técnica

Repositórios voltados a conhecimento funcional e documentação de processos.

- `linx-dms-knowledge-base`
- `dealership-business-processes`
- `erp-troubleshooting-handbook`
- `sql-for-support`

---

## Estrutura padrão para cada repositório

```text
README.md
ROADMAP.md
CHANGELOG.md
LICENSE
docs/
diagrams/
assets/
examples/
```

---

## Modelo de README para projetos

Cada README deve responder:

1. O que é o projeto?
2. Qual problema ele resolve?
3. Quais tecnologias usa?
4. Como rodar?
5. Como está organizado?
6. Qual o roadmap?
7. Quais aprendizados ele demonstra?

---

## Critério para destacar projetos

Um projeto só deve ser fixado no perfil se tiver pelo menos:

- README claro
- Objetivo definido
- Tecnologias listadas
- Prints, diagramas ou exemplos
- Status do projeto
- Próximos passos

---

## Posicionamento desejado

O GitHub deve transmitir a seguinte mensagem:

> Este profissional entende suporte, processos de negócio, sistemas corporativos, documentação, automação e desenvolvimento de soluções.
